# Prompt Design Workbench

A personal workbench tool for designing, testing, and iterating on LLM agent system prompts using the orthogonal prompt design framework.

**Status:** Production-ready | **270 Tests** | **92% Coverage** | **5-minute iteration cycle**

---

## What It Does

Transform agent specifications into system prompts in minutes. Test them live against the LLM. Refine based on real output. Repeat until satisfied.

```bash
# 1. Create a spec
promptctl init-spec my-analyst

# 2. Generate a prompt (LLM-powered)
promptctl design my-analyst.agent-spec.md

# 3. Test it live
promptctl run my-analyst --message "What are the top 3 risks?"

# 4. Refine based on results
promptctl refine my-analyst --feedback "Add JSON schema requirement"

# 5. Test again
promptctl run my-analyst --message "What are the top 3 risks?"
```

Done in under 5 minutes per iteration.

## Why This Workbench?

**Design Framework:** The orthogonal Task/Plan/Rule Blocks structure keeps prompts maintainable and composable.

**LLM-Powered Generation:** Not just mechanical scaffolding—OpenAI enriches your specs with substantive goals, criteria, and constraints.

**Fast Iteration:** See real behavior instantly. Refine based on output, not guesswork.

**Test Suite:** 270 tests, 92% coverage. Confidence that changes work.

**Externalized Content:** Rule blocks, specs, and designs live as files (tracked in Git).

## Quick Start

### Prerequisites

- Python 3.11+
- OpenAI API key: set `OPENAI_API_KEY` in the environment or in a `.env` file (see below).
- `uv` for dependency management (optional, but recommended)

**Local development:** Copy `.env.example` to `.env` or `.env.local` and set your API key. These files are gitignored; the app loads them automatically via `python-dotenv`.

### Installation

```bash
cd /Volumes/Dev_Projects/Superloop/_DOCS_RESEARCH_LEARNINGS

# Install dependencies
uv pip install -e . --no-build-isolation

# Or with pip:
# pip install -e .
```

### Create Your First Agent

```bash
# 1. Scaffold a spec
promptctl init-spec my-agent

# 2. Edit the spec (add role, inputs, outputs, constraints)
vim my-agent.agent-spec.md

# 3. Generate a prompt (uses OPENAI_API_KEY from .env if set)
promptctl design my-agent.agent-spec.md

# 4. Test it
promptctl run my-agent --message "your test question"

# 5. Refine as needed
promptctl refine my-agent --feedback "your feedback"
```

## Core Concepts

### The Four Commands

| Command | Purpose | Example |
|---------|---------|---------|
| `init-spec` | Scaffold a new agent spec | `promptctl init-spec my-agent` |
| `design` | Generate system prompt from spec (LLM-powered) | `promptctl design spec.md --tier 2` |
| `run` | Test prompt against real messages | `promptctl run my-agent --message "test"` |
| `refine` | Improve design based on feedback | `promptctl refine my-agent --feedback "..."` |
| `evaluate` | Run structured test scenarios | `promptctl evaluate my-agent --scenarios tests.json` |

### Orthogonal Design: Task, Plan, Rule Blocks

Every prompt is represented in three orthogonal parts:

- **Task (WHAT)** — Goals, acceptance criteria, constraints (generated by LLM from your spec)
- **Execution Plan (HOW)** — Ordered steps for Tier 2/3 (generated by LLM)
- **Rule Blocks (DETAILS)** — Schemas, formatting rules, guardrails (you edit these files)

### Complexity Tiers

- **Tier 1** — Task only (creative/exploratory tasks)
- **Tier 2** — Task + lightweight plan (structured reasoning)
- **Tier 3** — Full task/plan + rule blocks (deterministic JSON/HTML output)

The design command auto-selects; you can override with `--tier`.

## Architecture

### Layering (Strict Dependencies)

```
[CLI & Storage] (files & CLI)
    ↓
[Agents] (design, refinement, evaluation)
    ↓
[Patterns] (tier selection & building)
    ↓
[Models] (Pydantic data structures)
    ↓
[Providers] (OpenAI API)
```

**Critical Design Decisions:**

1. **Renderer purity** — Renderers output exactly what they receive; zero logic
2. **Rule block symbolism** — References resolved only at render time
3. **Agent statelessness** — No state stored on agents; state flows in/out
4. **One-way dependencies** — No upward imports

See CLAUDE.md for full architecture details.

## Model and Verbosity Control

### Default Model (GPT-5.2-mini)

The workbench now defaults to **gpt-5.2-mini**, with support for reasoning effort levels:

```bash
# Design with default gpt-5.2-mini at medium verbosity
promptctl design spec.md

# Use gpt-5.2-mini with high reasoning effort
promptctl design spec.md --model gpt-5.2-mini --verbosity high

# Fall back to gpt-4o
promptctl design spec.md --model gpt-4o

# Test with specific model and verbosity
promptctl run my-agent --message "test" --model gpt-5.4-pro --verbosity low
```

### Supported Models

- **gpt-5.2-mini** (default) — Lightweight, optimized for cost-efficient reasoning
- **gpt-5.4** — Latest full model, optimized for reasoning
- **gpt-5.4-pro** — Professional variant with enhanced capabilities
- **gpt-5-mini** — Lightweight variant
- **gpt-4o** — Previous generation (fallback compatible)

### Reasoning Effort Levels (GPT-5.x only)

When using gpt-5.x models, set reasoning effort with `--verbosity`:

- **low** — Fast, minimal reasoning
- **medium** (default) — Balanced
- **high** — Deep reasoning, longer processing

> **Note:** `--verbosity` only affects gpt-5.x models; it's silently ignored for gpt-4.x.

## Troubleshooting

### "ModuleNotFoundError: No module named 'typer'"
Dependencies not installed. Run: `uv pip install -e .`

### "OPENAI_API_KEY is not set"
Set your API key in the environment or in a `.env` file (copy from `.env.example`). The app loads `.env` and `.env.local` from the current working directory when you run `promptctl`.

### "LLM response was not valid JSON"
Check `--verbose` output. The LLM may have returned markdown fences. The tool falls back to heuristic mode gracefully.

### "Spec file not found"
Check your path. Specs live in `projects/<identifier>/specs/`.
