## Repository Purpose & Status

This is a **working software project**: a prompt engineering workbench tool that combines research-backed prompt design methodology with practical Python implementation.

**Status:** Production-ready
- ✅ Fully implemented (5 CLI commands, 3 agents)
- ✅ Tested (270 tests, 92% coverage)
- ✅ Documented (handbook, design docs, code comments)
- ✅ Ready for daily use

### What It Is

A command-line tool for designing system prompts using the orthogonal Task/Plan/Rule Blocks framework. Users create Markdown agent specs, the tool generates LLM-powered prompts, they test and refine in an iteration loop.

### What It's Not

- ❌ Not a library for others to use (it's personal/team-focused)
- ❌ Not cloud-hosted (runs locally)
- ❌ Not a general framework (specific to the orthogonal design approach)

```
[CLI] ← owns user interaction

  ↓
[Models] ← owns data structures
  ↓

```

**Critical Rules:**
- ✅ `models/` has ZERO internal imports (pure data)
- ✅ `patterns/` imports only `models/`
- ✅ `agents/` imports `models/` + `patterns/` + `rule_blocks/`
- ✅ `cli/` imports everything above (topmost)
- ❌ NEVER import upward (e.g., models should never import agents)

### External Files (`ai_components/`, `projects/`)

- **`ai_components/rule_blocks/`** — User-editable Markdown rule blocks (primary, highest priority)
- **`prompt_design_system/rule_blocks/`** — Package reference implementations (secondary, lowest priority)
- **`projects/`** — Auto-created by tool; user's agents live here (Git-tracked)

#### Rule Block Loading Order

`RuleBlockRegistry.from_config()` loads from both directories and merges the results:

1. `prompt_design_system/rule_blocks/` is loaded first (reference implementations shipped with the package).
2. `ai_components/rule_blocks/` is loaded second and its entries overwrite any same-named entry from step 1.

This means `ai_components/` is always authoritative.  Drop a `.md` file there with the same stem as a package block to override it.  Add a brand-new `.md` file there to extend the registry beyond what the package ships.  Never edit `prompt_design_system/rule_blocks/` directly — those files are reference implementations and will be overwritten on package upgrades.

## Core Concepts (For Development)

### Orthogonal Design (The Framework We Encode)

Three non-overlapping components:

1. **Task Block (WHAT)** — Goals, acceptance criteria, constraints
   - Generated by LLM from AgentSpec
   - Never contains execution steps or formatting rules

2. **Execution Plan Block (HOW)** — Ordered steps for multi-step reasoning
   - Generated by LLM (for Tier 2+)
   - Never contains formatting/schema details
   - References Rule Blocks by name

3. **Rule Blocks (DETAILS)** — Schemas, formatting, guardrails
   - Stored as Markdown files (user editable)
   - Single source of truth for a rule
   - Referenced by name from Task/Plan

**Key Invariant:** Zero duplication across components. If the same constraint appears in Task AND Plan, architecture is leaking.

### Tiered Complexity Model (How We Choose Structure)

| Tier | Structure | Use Case | Auto-Detection |
|------|-----------|----------|-----------------|
| **1** | Task only | Creative/open-ended | No structure keywords |
| **2** | Task + lightweight Plan | Structured reasoning | "constraint", "step", "structure" |
| **3** | Full Task + Plan + Rule Blocks | Deterministic output | "JSON", "schema", "format", "HTML" |

**Decision Logic** (in `PatternSelector`):
- Scan spec for keywords
- Default to Tier 1, escalate based on structure/determinism needs
- User can override with `--tier` flag

### Critical Design Contracts (Enforce These)

#### Contract 1: Renderer Purity

**Rule:** Renderers output exactly what they receive. Zero conditional logic.

```python
# ✅ GOOD
def render(design):
    output = render_task(design.task)
    if design.execution_plan:
        output += render_plan(design.execution_plan)
    return output

# ❌ BAD
def render(design):
    if design.tier == TIER_3:  # LOGIC IN RENDERER!
        # ... special handling
```

**Enforcement:** Code review + contract tests in `tests/test_*.py`

#### Contract 2: Rule Block Reference Symbolism

**Rule:** `RuleBlockRef` is a symbolic name reference (not resolved) until render time.

```python
# ✅ GOOD: Keep references symbolic
task_block.rule_block_refs = [RuleBlockRef(name="citations")]

# ❌ BAD: Early resolution
task_block.rule_blocks = [registry.get("citations")]
```

**Why:** Decouples design construction from registry availability.

#### Contract 3: Agent Statelessness

**Rule:** Agents are callable objects with configuration only. All state flows in/out via arguments.

```python
# ✅ GOOD
agent = DesignAgent(pattern_selector=..., llm_client=..., verbose=False)
design1 = agent.create_design(spec1)
design2 = agent.create_design(spec2)  # Independent call, no cross-contamination

# ❌ BAD: Agent holds design state
agent.design = design1
agent.refine()  # Mutates agent.design
```

#### Contract 4: Explicit Error Handling

**Rule:** Agents fallback gracefully. If LLM fails, return heuristic. Never silent failures.

```python
# ✅ GOOD
try:
    response = self.llm_client.generate(prompt)
except RuntimeError as exc:
    logger.warning("LLM failed; using heuristic. Reason: %s", exc)
    return design_heuristic

# ❌ BAD
try:
    response = self.llm_client.generate(prompt)
except:
    pass  # Silent failure; caller has no idea
```

## Development Workflow

### Adding a New Feature

**Example:** "Add a `list` command to show all saved agents"

1. **Write tests first** (TDD style)
   ```python
   # tests/test_cli.py
   def test_list_command_shows_all_agents(tmp_path):
       """Verify list command enumerates projects."""
       storage = ProjectStorage(config_with_root(tmp_path))
       # Create two agents
       storage.ensure_project_dirs("agent1")
       storage.ensure_project_dirs("agent2")
       # Call CLI
       runner = CliRunner()
       result = runner.invoke(app, ["list"])
       assert result.exit_code == 0
       assert "agent1" in result.stdout
       assert "agent2" in result.stdout
   ```

2. **Implement the feature**
   ```python
   # prompt_design_system/cli.py
   @app.command("list")
   def list_agents() -> None:
       """List all saved agent projects."""
       config = AppConfig.from_cwd()
       projects_dir = config.paths.projects_dir
       agents = sorted([d.name for d in projects_dir.iterdir() if d.is_dir()])
       for agent in agents:
           console.print(f"  {agent}")
   ```

3. **Verify tests pass**
   ```bash
   uv run --with pytest python3 -m pytest tests/test_cli.py::test_list_command_shows_all_agents -v
   ```

4. **Update documentation** (WORKBENCH_DESIGN.md or PROMPT_DESIGN_HANDBOOK.md)

5. **Commit with clear message**
   ```bash
   git add prompt_design_system/cli.py tests/test_cli.py
   git commit -m "add list command to enumerate saved agents"
   ```

## Key Implementation Details

### How Design Works (The Main Loop)

```python
# User runs: promptctl design spec.md --tier 2 --verbose

# 1. Parse Markdown spec
agent_spec = parse_agent_spec_from_markdown("spec.md")

# 2. Load rule blocks
registry = RuleBlockRegistry.from_config(config)

# 3. Create design agent with LLM
llm_client = OpenAIProvider(api_key=os.getenv("OPENAI_API_KEY"))
design_agent = DesignAgent(
    pattern_selector=PatternSelector(),
    rule_block_registry=registry,
    llm_client=llm_client,
    verbose=True  # Print LLM prompt
)

# 4. Create design (with LLM enrichment)
design = design_agent.create_design(agent_spec, explicit_tier="2")

# 5. Render to system prompt
rendered_prompt = _render_prompt(design)

# 6. Save
storage.save_design(design)
storage.save_rendered_prompt(design, rendered_prompt)
```

**What the LLM Does:**
- Receives AgentSpec (role, inputs, outputs, constraints)
- Receives selected tier (1, 2, or 3)
- Returns JSON with substantive Task Block:
  - `goal` — Clear, specific outcome for the agent
  - `acceptance_criteria` — 3-4 measurable conditions
  - `constraints` — 2-3 guardrail constraints
  - `execution_steps` — (for Tier 2+) concrete action steps

**What the Tool Does:**
- Parses LLM JSON response
- Gracefully falls back to heuristic if LLM fails
- Builds ExecutionPlan (for Tier 2+) with steps
- Resolves rule blocks by name from registry
- Renders complete system prompt

### How Refinement Works

```python
# User runs: promptctl refine my-agent --feedback "Add JSON requirement"

# 1. Load previous design
design = storage.load_design("my-agent")

# 2. Build refinement prompt (includes current design + feedback)
prompt = _build_refinement_llm_prompt(design, feedback)

# 3. Call LLM with refinement prompt
response = llm_client.generate(prompt)

# 4. Parse response (same schema as original)
refined_task, suggested_steps = _parse_llm_task_enrichment(
    response,
    fallback_task=design.task,
    wants_steps=design.tier in (TIER_2, TIER_3)
)

# 5. Rebuild design with refined task
refined_design = PromptDesign(
    agent_spec=design.agent_spec,
    tier=design.tier,
    task=refined_task,
    execution_plan=ExecutionPlanBlock(...) if steps else None,
    rule_blocks=design.rule_blocks
)

# 6. Save
storage.save_design(refined_design, overwrite=True)
```

## Anti-Patterns to Avoid

### Policy Echo
Stating the same constraint in multiple places (Task AND Plan AND Rule Block).

**Fix:** State once, reference elsewhere. Use Placement Router (PROMPT_DESIGN_HANDBOOK.md) to decide where.

### Mixed Intent
Task says "be concise", Plan says "be detailed". Conflicting signals to LLM.

**Fix:** Reconcile before passing to LLM. Usually move detail to Plan, concision to Task.

### Dual Ownership
Both Rule Block AND Task say "output as JSON".

**Fix:** Task says "follow Rule Block: json-schema". Rule Block owns the definition.

### Schema Leakage
Task describes JSON structure instead of referencing a Rule Block.

**Fix:** Task says "see Rule Block: person-schema". Keep schema in Rule Block only.

### Renderer Logic Creep
Adding conditional logic to renderers (checking tier, checking rule block presence).

**Fix:** If a choice is needed, make it upstream (in patterns/). Renderers are mechanical only.

## Common Questions

**Q: Where should I add validation logic?**
A: In parsers (for input validation) and agents (for output validation). Models are pure data.

**Q: How do I add support for Anthropic API?**
A: Create `AnthropicProvider` in `providers.py` (inherits from `LLMClient` Protocol). Wire it in `cli.py`'s `_resolve_llm_client()`. No other changes needed.

**Q: Can I change the LLM prompt?**
A: Yes, but carefully. The prompt is in `agents.py` as `_build_llm_prompt()`. Changes here affect all design generation. Test thoroughly.

**Q: How do I add a new CLI command?**
A: Create a function with `@app.command("name")` in `cli.py`. Write tests in `tests/test_cli.py` first. Follow the patterns in existing commands.

**Q: What if the LLM returns invalid JSON?**
A: Caught in `_parse_llm_task_enrichment()`. Falls back to heuristic. Check `--verbose` output to debug.

**Q: Can I version rule blocks?**
A: Currently no. Rule blocks are edited in place (Git tracks versions). Multi-version support would require registry changes.

## Summary: Golden Rules

1. **Tests first** — Write tests before code
2. **Orthogonality always** — Zero duplication across Task/Plan/Rule Blocks
3. **Contracts matter** — Renderer purity, reference symbolism, statelessness
4. **Backwards compatible** — Don't break existing prompts/designs
5. **Graceful degradation** — If LLM fails, fall back to heuristic
6. **Document as you go** — Update WORKBENCH_DESIGN.md or HANDBOOK.md when adding features
7. **Dependency hygiene** — Maintain strict one-way dependency flow

**When in doubt, ask:** "Does this violate the orthogonal principle?"
